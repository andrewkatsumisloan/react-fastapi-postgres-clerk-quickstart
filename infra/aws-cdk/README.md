# AWS CDK Deployment

This is the recommended AWS IaC path for the starter. It deploys the React app
as static assets behind CloudFront, deploys the FastAPI app to ECS Fargate, and
creates private RDS PostgreSQL.

## Architecture

```text
Browser
  |
  v
CloudFront
  |-- /        -> private S3 bucket with Vite build output
  |-- /api/*   -> Application Load Balancer -> ECS Fargate -> RDS PostgreSQL
```

Secrets:

- RDS credentials are generated by CDK and stored in Secrets Manager.
- Clerk backend secret is created as `/<appName>-<stage>/clerk` with a
  placeholder value. Replace it after the first deploy.

Migrations:

- CDK creates a dedicated Fargate task definition that runs
  `alembic upgrade head`.
- Use `scripts/run-migrations.sh` after deploys that include schema changes.

## Prerequisites

- AWS CLI authenticated to the target account.
- AWS CDK bootstrapped in the target account and region.
- Docker running locally so CDK can build the API image asset.
- Node 20+.
- A Clerk application.

Bootstrap once per account/region:

```bash
cd infra/aws-cdk
npm install
npx cdk bootstrap
```

## First Deploy

Build and deploy from this directory:

```bash
cd infra/aws-cdk
npm install

VITE_CLERK_PUBLISHABLE_KEY=pk_live_... \
  npm run deploy -- \
  -c appName=fullstack-starter \
  -c stage=dev \
  -c clerkJwtIssuer=https://your-clerk-instance.clerk.accounts.dev
```

After the stack creates the placeholder Clerk secret, replace it:

```bash
aws secretsmanager put-secret-value \
  --secret-id /fullstack-starter-dev/clerk \
  --secret-string '{"CLERK_SECRET_KEY":"sk_live_..."}'
```

Run database migrations:

```bash
AWS_REGION=us-east-1 ./scripts/run-migrations.sh fullstack-starter-dev
```

Then open the `CloudFrontUrl` output from the CDK deploy.

## Useful Context Values

All values are optional except the Clerk issuer for authenticated API usage.

```bash
-c appName=fullstack-starter
-c stage=dev
-c clerkJwtIssuer=https://your-clerk-instance.clerk.accounts.dev
-c clerkAudience=your-audience
-c corsAllowOrigins=https://your-domain.example.com
-c dbName=app
-c dbUsername=appuser
-c desiredCount=1
```

For same-origin browser traffic through CloudFront, `corsAllowOrigins` can stay
empty because requests go to `/api/*` on the same CloudFront domain.

## Frontend Configuration

The deploy script runs:

```bash
cd ../../client
npm ci
npm run build
```

Pass Vite build-time variables before `npm run deploy`, especially:

```bash
VITE_CLERK_PUBLISHABLE_KEY=pk_live_...
```

Leave `VITE_API_BASE_URL` unset for the default same-origin CloudFront routing.
The client falls back to `window.location.origin`, and CloudFront proxies
`/api/*` to the API load balancer.

## Updating The App

For frontend-only changes:

```bash
VITE_CLERK_PUBLISHABLE_KEY=pk_live_... npm run deploy -- -c stage=dev
```

For backend changes:

```bash
VITE_CLERK_PUBLISHABLE_KEY=pk_live_... npm run deploy -- -c stage=dev
```

For schema changes:

```bash
AWS_REGION=us-east-1 ./scripts/run-migrations.sh fullstack-starter-dev
```

## Production Notes

The starter defaults are deliberately modest:

- One Fargate task.
- `t3.micro` RDS.
- Public-subnet Fargate tasks to avoid NAT Gateway cost.
- CloudFront HTTPS for the public app URL.
- RDS deletion protection only when `stage=prod`.

Before using this for a serious production app, consider:

- Custom domain and ACM certificate on CloudFront.
- WAF on CloudFront.
- Private Fargate subnets with NAT Gateway or VPC endpoints.
- RDS Multi-AZ, larger instance class, and longer backup retention.
- GitHub Actions OIDC deployment role.
- Moving the Clerk secret from placeholder-created to pre-created/imported.

## Destroying A Non-Production Stack

```bash
npm run destroy -- -c stage=dev
```

For `stage=prod`, RDS is configured for snapshot-on-delete and the site bucket is
retained. Expect manual cleanup for protected production data.
